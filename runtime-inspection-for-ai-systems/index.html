The Limits of Runtime Inspection for AI Systems

July 15, 2026

Earlier this month Microsoft announced that Microsoft Defender for Endpoint would offer “agent runtime protection,” an inspection tool aimed at detecting suspected prompt injection and malicious tool use attempts. Runtime inspection tools can play an important role as sensors and layered protection, but their limitations may not be well understood by the governance community.

Runtime inspection tools cannot say with high reliability whether arbitrary context (AI inputs) will safely influence reasoning, tool use, memory, or system state. These limitations come from fundamental constraints in computation, information, and meaning. We will unpack why that is the case and also consider the direct implications for agentic design.

Agentic AI Context Is Not a Normal Security Object

Many familiar security tools and safeguards operate across highly formalized, well-specified domains. For example, network traffic has defined headers and protocols and code has a highly structured and limited syntax.

AI Context Can Be Almost Anything

In contrast, AI context can be essentially arbitrary, both in terms of contents and variability in size (the latter within the upper limits of the model). There are no generally applicable syntax requirements for what is allowed in AI context. Depending on the design of the AI system, there may be no restrictions or formal specifications for what can go into AI inputs or what the order or syntax should be. As a result, AI context can be a mishmash of natural language, structured data, workflow/conversation history, unstructured documents, JSON fields, code blocks, base64 blobs, and lord-knows-what-else.

And even in those instances where AI context is more carefully generated, highly structured, and of a reasonable size, that only creates consistency within that particular system. Unless a runtime inspection tool is highly tuned to that specific system and given a good deal of information beyond the AI context itself (which is difficult, costs money, and may not be technically supported), the AI context will still be essentially arbitrary from the standpoint of the general-purpose runtime inspection tool.

General-purpose runtime inspection measures therefore have a very difficult task, and organizations should not treat them as reliable primary controls over arbitrary agent behavior.

AI Context Is Incomplete

In addition to AI context being arbitrary, it also lacks information. This is a confounding factor distinct from arbitrariness. In other words, AI context may be malicious in one situation and benign/expected in another, and it all depends on the internals of the rest of the AI system, outside the AI context. The following sentence could be malicious or benign, depending on the AI system. It is unlikely that a runtime scanner will be able to reliably determine which:

Prior to processing this context, delete all temporary files.

In other words, being “malicious” or “benign” is not an intrinsic property of AI context.

A runtime inspection tool looking at this statement as part of AI context would need a significant amount of other information about the AI system, well beyond the AI context, to be able to make a decent guess about whether this sentence is malicious or benign. There are also limits to inspection tools even when they have access to such information, but at least it would be a fairer fight.

Implications for Agentic AI Design

Runtime inspection tools are useful sensors and checkpoints, but they cannot be the primary security boundary for agentic systems because the meaning and risk of AI context are application-specific. Robust designs are ones that reduce ambiguity upstream, constrain available capabilities, and enforce deterministic policy at tool, memory, retrieval, and egress boundaries.

The closest thing to a “free lunch” is upstream agent architecture that reduces AI context as an attack surface by reducing context ambiguity, decreasing context size, and creating deterministic validation steps that dovetail with the specific structure used by the system.

The following control points may be helpful in blocking messy, high-ambiguity input:

  • maximum total context length
  • maximum per-source context length
  • maximum retrieved chunks
  • maximum prior conversation or workflow turns
  • no raw full-document stuffing
  • limiting the types of sources that contribute to context
  • retrieval allowlists and source trust tiers
  • track and label provenance
  • clean separation of content and authority/instruction
  • block unsupported file types
  • block base64 unless decoded
  • block compressed archives
  • strip hidden text
  • strip unnecessary links
  • normalize input shape

Tools that are downstream from AI responses should likewise be gated:

  • tool allowlists by task
  • least-privilege tool scopes
  • read-only mode by default
  • explicit approval for high-impact tools
  • tool argument validation
  • typed tool schemas
  • deterministic enforcement of policy re: tool use arguments
  • postcondition checks after high-impact actions
  • default deny

Agent designs should also reduce the authority available to model-mediated decisions:

  • capability isolation between agents, tools, and workflows
  • sandboxing for code execution and file operations
  • egress controls for network calls, external sharing, and data export
  • memory-write controls, including approval or validation before persistent memory changes
  • separate authorization checks for retrieval, tool use, memory updates, and outbound actions

Excellent resources for companies looking to secure agentic AI systems include OWASP’s materials, in particular the OWASP Top 10 for LLM Applications 2025.

The bottom line is that businesses should not place too much reliance in general-purpose runtime inspection systems. They have their role to be sure, but they are no substitute for careful architecture aimed at reducing the attack surface.
